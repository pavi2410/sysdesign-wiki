## What it is

Publish/subscribe is an event distribution pattern where publishers send events to topics and subscribers receive events from topics they care about.

## Why it matters

It decouples event producers from consumers. A single message can update WebSocket clients, invalidate caches, trigger analytics, and notify downstream systems without the producer knowing each one.

## How it works

Events are published to channels, topics, or streams. Subscribers register interest and receive matching events. Some systems retain events for replay; others only deliver live messages.

Each subscriber processes the event independently and at its own pace. A slow subscriber does not block the others, though it may fall behind if the system does not buffer or retain events.

## When to use it

Use pub/sub for live updates, notifications, cache invalidation, analytics pipelines, and service-to-service event distribution.

## Tradeoffs / failure modes

Delivery guarantees vary. Ordering, replay, slow subscribers, and duplicate events must be understood before building critical workflows on top.

## Tiny example

Slack publishes `message.created` to a channel topic so connected clients, search indexing, and notifications can react independently.
